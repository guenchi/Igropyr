# Keep-Alive 热路径优化报告

日期：2026-07-10
分支：`agent/optimize-keepalive-hot-path`

## 要求

本次优化以 pooled Express keep-alive 吞吐为主指标，保持 crash retry、stuck-kill、请求隔离及既有公开 HTTP API。不引入 Scheme bytevector 的异步零拷贝；新增的公开接口仅为 `(tcp-writev! conn chunks on-done)`。

验收门槛：主场景 `ab -k -n 300000 -c 500` 的吞吐中位数提升不少于 8%、每请求分配下降不少于 20%；四个辅助场景不得回退超过 5%；每个版本 7 轮的 CV 不超过 5%，且所有请求零失败。

## 已完成

### 热路径实现

- HTTP reader 改为私有 cursor buffer：空 unread 时直接接管 read bytevector，跨 read 时几何扩容，只移动消费游标，不再复制 leftover。
- header terminator 从上次扫描位置的三字节重叠处继续扫描；保留 8 KiB header/trailer、1 MiB body 和 pipeline 上限。
- 普通固定长度 body 仅复制一次；空 body 复用全局空 bytevector。
- request line 与 header 改为单遍 bytevector parser；常用 method/header 使用既有 symbol，最终 path/query/header value 才转为字符串；无 `%`/`+` 的 URL 直接返回原字符串。
- `(igropyr uv)` 新增 `tcp-writev!`，兼容 vector 和 proper list，回调在提前关闭、立即写失败与正常完成时均恰好一次。
- 普通响应、流式 chunk 分别使用 gather-write；流式 framing 避免嵌套 `bv-append`。
- 响应头使用一项有界 MRU：命中时复用已编码 header；缓存键复制 header 字符串，避免调用方后续修改字符串产生陈旧响应。
- Express 没有 fast route 时不再为每个 pooled 请求拆分一次 path；监听后新增 fast route 仍可见。

### 基准、采样与文档

- 新增 `bench/`：Express pooled、core pooled、Express fast 三种最小 server，支持 source/compiled A/B、交替 7 轮、warm-up、CV 与 gate 检查。
- 输出 req/s、p95、RSS、CPU、GC 次数、GC CPU/real time、GC 扫描字节和 allocated bytes/request；实际完成请求数必须等于 `ab -n`。
- 新连接场景支持递增端口与可配置的 `NEW_CONN_COOLDOWN=30`，降低 macOS TIME_WAIT 对后续轮次的污染。
- 提供 Scheme `compile-profile` 构建以及 macOS `sample` 工作流；新增中英文 benchmark 文档。

### 回归覆盖

- request line/header terminator/Content-Length/chunked body 的逐字节拆包。
- 同 read 多个 pipeline 请求、handler 等待期间继续接收 pipeline 数据、cursor compact 和 pipeline 上限关闭。
- `tcp-writev!` 多段、空段、大 body、关闭连接 callback 恰好一次。
- streaming chunk framing、可变 header 字符串缓存正确性、连接释放检查。

## 未完成或有意调整

- `new-500` 与 `new-1000` 未获得双方 CV 不超过 5% 的有效 7 轮结果。多次重跑均为 0 failed，但本机在高并发新连接下出现明显内核/调度波动；因此不能宣称已通过完整辅助场景 gate，也不应据不稳定中位数判断真实回退。
- source-loaded 主场景显示约 +22% 至 +25% 吞吐与约 -61% 分配，但 baseline CV 多次超过 5%，仅作为方向复核，未作为正式 gate 结论。
- native `sample` 显示 `malloc/free` 不到 1% 样本，未达到 10% 门槛，故未实现 foreign write block slab。
- actor 私有协议压缩曾试验，未显示稳定增益，已回退；未改变 supervisor、retry、stuck-kill 或一连接一 reader 模型。
- 未实施 `uv_tcp_t` memory reuse；新连接优化按计划不进入本 PR。
- 原计划中的 `tcp-writev!` 多 iovec 提交调整为：各 Scheme segment 仍复制到同一块最终 foreign block，但因 payload 相邻，提交给 libuv 时合并成一个 `uv_buf_t`。这是根据 native sample 中 `writev` 系统调用占比高作出的有意调整，避免重新引入 Scheme 中间拼接，同时减少 iovec syscall 开销。
- 由于完整辅助场景 gate 尚未通过，本次不创建 PR；后续应在低后台负载、可控网络栈环境中重跑 `new-500` 和 `new-1000`。
